## Problem Statement

두 팀 또는 두 페어가 같은 Pintos 목표를 각자 구현한 뒤, merge 담당자가 어느 코드가 더 좋은지 비교하고 하나 또는 둘의 코드를 섞어 `dev`로 올릴 PR을 만들고 싶다.

현재 GitHub compare와 일반 line diff는 "몇 번째 줄이 달라졌는지"는 보여주지만, merge 담당자가 실제로 판단해야 하는 단위인 함수, 구조체, helper 흐름을 먼저 보여주지 않는다. 예시 비교인 `woojin@c8d2936` vs `pair/heejun-donghyun@690d8a9`에서는 변경 파일이 `pintos/userprog/process.c`, `pintos/vm/vm.c`에 집중되고, 핵심 symbol은 `lazy_load_segment`, `load_segment`, `vm_alloc_page_with_initializer`다.

## What Makes This Cool

이 앱은 "예쁜 diff viewer"가 아니라 merge 담당자의 판단 보조 장치다. 화면 첫 줄에서 `load_segment`가 바뀌었고, 그 안에서 `ofs += page_read_bytes` 같은 동작상 중요한 차이가 있다는 것을 바로 보여준다. 리뷰어는 전체 파일을 스크롤하기 전에 어떤 함수가 위험하고 어떤 함수가 안전한지 볼 수 있다.

## Constraints

- 1차 사용자는 `week11-team-01-pintos-vm`의 merge 담당자다.
- 1차 언어는 C다. PintOS 과제 특성상 `pintos/userprog`, `pintos/vm`, `pintos/threads`가 우선이다.
- `dev`는 GitHub ruleset상 PR이 필요하고 승인 2개가 필요하다. 앱은 `dev`로 직접 merge하지 않는다.
- MVP 산출물은 symbol diff 검토 기록과 PR 초안이다.
- 자동 code merge는 MVP의 중심이 아니다. 잘못된 자동 혼합은 팀의 학습과 안전을 해친다.

## Premises

1. merge 담당자는 파일 단위보다 함수/구조체 단위로 판단한다.
2. semantic diff engine이 제품의 중심이고, GitHub PR 생성은 후속 전달 경로다.
3. C parser는 Tree-sitter C grammar를 사용한다.
4. line diff는 버리지 않는다. symbol diff에서 관심 지점을 고른 뒤, split view와 inline diff로 내려간다.
5. 앱은 결정 로그를 남긴다. 예: `load_segment: pair/heejun-donghyun 채택`, `vm_alloc_page_with_initializer: woojin 채택`, `lazy_load_segment: 수동 검토 필요`.

## Landscape Notes

- Tree-sitter는 syntax node의 byte/row/column 위치를 제공한다. 따라서 symbol index는 함수명, 파일 경로, 시작/끝 위치를 안정적으로 가질 수 있다. Reference: [Tree-sitter Basic Parsing](https://tree-sitter.github.io/tree-sitter/using-parsers/2-basic-parsing.html).
- Tree-sitter query는 syntax tree 안에서 특정 node pattern을 찾는 방식이다. C 함수 정의, struct 정의, typedef, macro 영역을 query로 뽑기에 적합하다. Reference: [Tree-sitter Query Syntax](https://tree-sitter.github.io/tree-sitter/using-parsers/queries/1-syntax.html), [tree-sitter-c](https://github.com/tree-sitter/tree-sitter-c).
- Difftastic은 Tree-sitter 기반 syntax-aware diff를 검증한 좋은 선례다. line diff보다 "실제로 바뀐 syntax"를 읽기 쉽게 보여주는 것이 핵심 가치다. Reference: [Difftastic](https://difftastic.wilfred.me.uk/).
- SemanticMerge가 주는 교훈은 method 단위 이동/충돌을 제품의 중심 개념으로 잡아야 한다는 점이다. 하지만 MVP에서 자동 semantic merge까지 가면 위험하다. Reference: [SemanticMerge features](https://www.semanticmerge.com/features).
- GitHub REST API는 compare, PR 생성, PR 파일 목록, PR review endpoint를 제공한다. 단, GitHub compare는 commit reachability 관점이 강해서 "두 commit의 tree 상태 차이"를 보는 source of truth로 쓰면 안 된다. 예시 commit pair에서도 GitHub compare는 merge-base 관계 때문에 파일 목록을 비워 줄 수 있다. Semantic diff의 기준은 local `git diff <left> <right>`와 `git show <sha>:<path>`다. GitHub는 metadata, ruleset, PR 생성에 쓴다. Reference: [GitHub Compare Commits](https://docs.github.com/en/rest/commits/commits), [GitHub Pull Requests API](https://docs.github.com/en/rest/pulls).
- CodeMirror merge view는 split merge UI의 현실적인 후보이고, original document update effect를 제공한다. Reference: [CodeMirror merge package](https://github.com/codemirror/merge), [CodeMirror Reference](https://codemirror.net/docs/ref/).

## Approaches Considered

### Approach A: GitHub-first 리뷰 콘솔

GitHub API로 브랜치, 커밋, PR, ruleset을 읽고, local git clone은 diff 분석용 캐시로만 쓴다. UI에서 symbol별 결정을 남기고, GitHub PR 초안을 만든다.

Pros:
- `dev` ruleset과 PR 승인 흐름에 가장 자연스럽다.
- 첫 버전에서 "결정 로그가 붙은 PR"까지 빠르게 도달한다.
- GitHub compare URL, PR 목록, branch 상태를 그대로 활용할 수 있다.

Cons:
- 실제 제품의 차별점인 semantic diff가 부속 기능처럼 밀릴 수 있다.
- 코드 혼합은 처음에 manual assist가 된다.

### Approach B: Local worktree merge assistant

앱이 repo를 로컬 worktree로 관리하고, 선택한 symbol을 실제 파일에 적용해 `merge-candidate/*` 브랜치를 만든다. 이후 테스트를 돌리고 GitHub PR을 생성한다.

Pros:
- 선택한 코드가 실제 브랜치에 반영되는 경험이 강하다.
- 테스트 결과까지 PR에 붙이는 end-to-end 흐름으로 확장하기 좋다.

Cons:
- 함수 일부만 섞을 때 patch conflict와 잘못된 적용 리스크가 크다.
- 로컬 git 상태, dirty worktree, 컨테이너 테스트까지 MVP에 들어온다.

### Approach C: Symbol Diff Engine First

Tree-sitter 기반 C symbol parser와 diff model을 먼저 만든다. GUI는 symbol explorer, split diff, inline diff, decision log에 집중한다. PR 생성은 결정 로그 export와 GitHub draft PR 생성으로 시작한다.

Pros:
- `lazy_load_segment`, `load_segment` 같은 함수 단위 비교 품질이 가장 좋다.
- 나중에 C 외 언어로 확장하기 쉽다.
- "줄 diff가 아니라 의미 단위 diff"라는 제품 가치를 먼저 검증한다.

Cons:
- PR 생성/브랜치 생성 자동화는 A보다 늦게 붙는다.
- parser, query, symbol matching 문제에 빠질 수 있다.
- 실제 코드 혼합은 Phase 2로 밀린다.

## Recommended Approach

Approach C를 중심으로 간다. 단, PR 초안 생성은 MVP에 포함한다.

제품의 핵심 데이터 모델은 `SymbolDiff`다.

```text
SymbolDiff
- symbol_id: stable key, e.g. "pintos/userprog/process.c::load_segment"
- kind: function | struct | typedef | macro | global
- status: unchanged | modified | added_left | added_right | deleted | moved | renamed | parse_error
- left_range: file, start_line, end_line, sha
- right_range: file, start_line, end_line, sha
- line_diff: inline hunks inside this symbol
- risk_flags: missing_cleanup | offset_update_changed | allocation_error_path | permission_change | parse_error
- decision: left | right | manual_mix | defer
- note: reviewer note for PR body
```

MVP UI는 4개 영역으로 구성한다.

1. Header: repo, base branch, left commit, right commit, compare URL, PR target.
2. Symbol sidebar: changed symbols grouped by file and risk.
3. Diff workspace: split view and inline view toggle.
4. Decision panel: choose left/right/manual/defer, write note, export PR draft.

## Key Flow

1. 사용자가 repo와 두 commit/branch를 선택한다.
2. 앱이 local repo cache에서 `git diff --name-status <left> <right>`로 변경 파일을 찾는다.
3. 앱이 `git show <sha>:<path>`로 변경 파일의 양쪽 원문을 읽는다.
4. Tree-sitter C parser가 함수/구조체/macro symbol index를 만든다.
5. symbol matcher가 같은 symbol, 추가된 symbol, 삭제된 symbol, moved candidate를 계산한다.
6. 각 symbol 내부에서 line diff를 계산하고, formatting-only 변화는 낮은 우선순위로 표시한다.
7. 사용자가 symbol별 결정을 남긴다.
8. 앱이 결정 로그와 compare 링크를 포함한 PR body 초안을 만든다.
9. 선택적으로 `merge-candidate/<date>-<topic>` 브랜치를 만들 수 있지만, MVP에서는 "PR 초안 생성"이 기본이다.

## Parser Edge Cases

C 코드는 전처리기 때문에 단순하지 않다. PintOS도 `#ifdef VM` 같은 조건부 컴파일 블록을 많이 쓴다. MVP는 전처리기를 실행하지 않고 raw source를 Tree-sitter로 파싱한다.

Rules:
- `#ifdef`, `#ifndef`, `#else`, `#endif`는 macro/preproc symbol로 sidebar에 표시한다.
- 함수가 조건부 컴파일 블록 안에 있으면 `condition: VM` 같은 context label을 붙인다.
- Tree-sitter `ERROR` 또는 `MISSING` node가 발견되면 해당 파일을 `parse_error`로 표시하고 file-level line diff로 fallback한다.
- macro body가 함수처럼 보이는 경우에는 함수 symbol로 승격하지 않는다.
- Phase 1은 rename/move를 "candidate"로만 표시한다. 자동 확정하지 않는다.

### Information Architecture

Use the approved 3-pane workbench as the first screen. The visual priority is:

1. **Changed symbol risk**: the left sidebar shows changed symbols grouped by file, with high-risk items visibly marked before low-risk items.
2. **Selected symbol diff**: the center workspace shows the currently selected symbol's split diff, focused on the smallest meaningful code range.
3. **Decision and PR note**: the right panel shows the selected symbol's decision state, action buttons, reviewer note, and PR body preview.

```text
+--------------------------------------------------------------------------------+
| repo / target branch / left ref / right ref                         PR actions |
+------------------------------+-------------------------------+-----------------+
| Changed symbols              | Selected symbol diff          | Decision        |
| - file grouping              | - split view by default       | - left/right    |
| - risk-first ordering        | - inline toggle available     | - manual/defer  |
| - search and risk filters    | - code range jump anchors     | - reviewer note |
+------------------------------+-------------------------------+-----------------+
```

The header is orientation, not navigation. The primary user action remains inside the selected symbol row and the decision panel.

### Interaction State Coverage

Every state must describe what the user sees, not only what the backend returns.

| Feature | Loading | Empty | Error | Success | Partial |
|---------|---------|-------|-------|---------|---------|
| Repo/ref selection | Show compact "비교 준비 중" status in the header and disable PR actions. | Show a short Korean empty state: "비교할 ref를 선택하세요" with left/right ref inputs focused. | Show the failing ref and a retry action; keep the last valid refs visible. | Header shows repo, target branch, left ref, right ref, and changed file count. | If one ref resolves and the other fails, keep the resolved ref and highlight only the failing input. |
| Changed symbol sidebar | Show skeleton rows shaped like symbol rows, not generic spinners. | Show "변경된 C 심볼이 없습니다" and offer file-level diff view. | Show file-level fallback rows with `parse_error` chips. | Show file groups sorted by risk, then file path, then symbol name. | Show parsed symbols first and a "건너뛴 파일" group for binary, too-large, or unsupported files. |
| Diff workspace | Show selected symbol title and code skeleton while loading. | Show "왼쪽에서 심볼을 선택하세요" with no decorative illustration. | Show the failing symbol range and a "파일 diff로 보기" fallback. | Show split diff by default, with inline toggle available. | If symbol extraction fails for one side, show available side plus file-level hunks. |
| Decision panel | Disable decision buttons until a symbol is selected. | Show "결정할 심볼을 선택하세요" and hide PR note preview. | Preserve typed note locally and show a non-destructive retry message. | Show selected decision, reviewer note, and PR body preview. | If some symbols are undecided, show remaining count and allow export with warnings. |
| PR export/draft | Show "PR 본문 생성 중" and keep the review workspace usable. | If no decisions exist, show "먼저 하나 이상의 심볼 결정을 남기세요". | For missing GitHub auth or protected branch issues, show the exact blocker and keep copy-to-clipboard export available. | Show draft PR URL or copied PR body confirmation. | If GitHub draft creation fails after PR body generation, keep the generated body and explain the manual next step. |

Empty states must be calm and useful. They should provide context plus the next concrete action, not decorative filler.

### User Journey And Emotional Arc

The design goal is to reduce merge anxiety by making risk visible, bounded, and recordable.

| Step | User does | User feels | UI support |
|------|-----------|------------|------------|
| 1 | Opens the comparison workspace with two refs loaded. | "I need to find the risky changes fast." | Header confirms repo/refs/target branch; sidebar immediately shows changed symbol count and high-risk rows. |
| 2 | Scans changed symbols by file and risk. | "The app has narrowed the problem for me." | Risk-first ordering, Korean labels, and stable C identifiers make the next click obvious. |
| 3 | Selects `load_segment`. | "This is the important part; show me the exact difference." | Center pane jumps to the function range and highlights behavior-changing lines such as `ofs += page_read_bytes`. |
| 4 | Chooses left/right/manual/defer and writes a note. | "My reasoning will not be lost." | Decision panel keeps the selected symbol, decision buttons, note field, and PR preview in the same visual column. |
| 5 | Exports or creates a draft PR. | "Reviewers can follow my reasoning without rediscovering everything." | PR body preview includes decisions, risk flags, skipped files, parse fallbacks, and reviewer notes. |

Time horizon:

- First 5 seconds: the user must see repo/refs, changed symbol count, and the highest-risk symbol without reading instructions.
- First 5 minutes: the user must complete at least one symbol decision and see it reflected in the PR body preview.
- Long-term trust: the app must never hide uncertainty. Parse errors, skipped files, undecided symbols, and GitHub auth failures stay visible and recoverable.

### Visual Guardrails

Classify this product as **App UI**, not a marketing or landing page.

Required visual direction:

- Use a calm, dense, readable workbench layout. The interface should feel closer to a focused code review console than a SaaS dashboard.
- Use Korean utility labels for UI controls and status text. Keep code identifiers, file paths, branch names, commit hashes, and risk flag enum names in source form.
- Prefer real layout regions over decorative cards. Cards are allowed only for repeated symbol rows, modals, and bounded tool panels.
- Use one primary accent for selection and focus. Risk colors are semantic only: high risk, warning, success, neutral.
- Body/UI type must support Korean well. Recommended default: `Pretendard` or `Noto Sans KR` for UI, `JetBrains Mono` or `D2Coding` for code.
- Code text must stay visually distinct from UI labels and must preserve monospace alignment.

Rejected patterns:

- No landing page, hero section, marketing copy, 3-column feature grid, testimonials, or decorative onboarding.
- No dashboard-card mosaic as the first screen.
- No purple/blue gradient theme, decorative blobs, icon circles, or ornamental illustrations.
- No centered-everything layout. The workspace should be scan-first and left-to-right.
- No placeholder-only labels in forms. Ref inputs and note fields need persistent labels.

Spacing and sizing:

- Use an 8px spacing grid.

- Text inside buttons and chips must fit at Korean label lengths without clipping.

### Responsive And Accessibility Requirements

Viewport behavior:

- Desktop `>= 1200px`: use the approved 3-pane layout: symbol sidebar, diff workspace, decision panel.
- Tablet `768px - 1199px`: keep symbol sidebar visible, move decision panel into a right-side drawer opened by a persistent "결정" button.
- Mobile `< 768px`: use a three-step tab flow: `심볼` -> `Diff` -> `결정`. The selected symbol name and decision state stay pinned at the top.
- Very narrow mobile `< 390px`: hide low-priority metadata behind a details disclosure; never hide the selected symbol, risk, or decision state.

Keyboard behavior:

- `j`/`k` moves between changed symbols.
- `Enter` opens the selected symbol diff.
- `1`, `2`, `3`, `4` choose left, right, manual, defer when the decision panel is focused.
- `Cmd/Ctrl+Enter` exports the PR body when required decisions are complete.
- All shortcuts must have visible menu labels or tooltip text; shortcuts are accelerators, not the only path.

Accessibility:

- Use landmarks for header, symbol navigation, main diff workspace, and decision aside.
- Symbol rows are keyboard-focusable and expose symbol name, file, status, and risk to screen readers.
- Risk must not rely on color alone. Include text labels such as `고위험`, `주의`, `변경됨`, or enum names.
- Body text contrast must meet WCAG AA. Code, chips, and disabled controls must remain readable.
- Touch targets for buttons, tabs, and symbol rows are at least 44px high on touch layouts.
- Diff additions/deletions must include non-color cues, such as `+`/`-`, labels, or line annotations.

### Resolved Design Decisions

| Decision | Resolution | Rationale |
|----------|------------|-----------|
| First-screen layout | Use the approved A-KO 3-pane workbench. | It keeps risk scanning, code inspection, and decision recording in one continuous flow. |
| UI language | Korean UI labels with source-form code identifiers. | Merge reviewers get local-language orientation without corrupting code terms. |
| PR export gate | Require every changed symbol to have `left`, `right`, `manual_mix`, or `defer` before enabling PR export. | `defer` handles uncertainty while preventing silent omissions from the PR body. |
| Mobile layout | Use tab flow: `심볼` -> `Diff` -> `결정`. | A 3-pane desktop layout cannot simply stack without burying decisions. |

### Not In Scope

- Automatic merge application to `dev`: deferred because branch protection and learning safety matter more than speed.
- Full visual design system for the entire repo: deferred until there are multiple UI surfaces beyond this tool.
- Patch skeleton generation for `manual_mix`: deferred because incorrect partial patches would create false confidence.
- 3+ implementation comparison UI: deferred because the MVP must first prove two-ref symbol comparison quality.

## Non-Goals

- No automatic merge to `dev`.
- No "AI decides winner" in MVP.
- No support for every language in Phase 1.
- No full three-way semantic merge in Phase 1.
- No branch protection bypass.

## Success Criteria

- Given the two example commits, the app lists the three modified symbols above without manual file inspection.
- Clicking `load_segment` jumps directly to that function's diff in split view.
- Reviewer can mark each symbol as left/right/manual/defer.
- Exported PR body includes repo, left/right commits, changed symbols, decisions, risk flags, and reviewer notes.
- The app may link to GitHub compare for context, but changed file detection comes from local `git diff <left> <right>`.
- The app can create a draft PR against `dev` when credentials are present and the user confirms.
- For parse errors, the app falls back to file-level line diff and marks the file as `parse_error`.
